## 1. 总体架构概览

### 1.1 设计分层

```
┌─────────────────────────────────────────────────────────┐
│  L3 云端生成层 (1-3s, 异步)                              │
│  LLM 对话 / 故事分支 / 离线预生成嵌入                      │
├─────────────────────────────────────────────────────────┤
│  L2 统计启发层 (<20ms, 异步)                             │
│  难度自适应 / 语义推荐 / 故事分支选择                      │
├─────────────────────────────────────────────────────────┤
│  L1 实时规则层 (<3ms, 同步)                              │
│  规则引擎 / 标签匹配 / 冲突检测 / 效用评估                 │
└─────────────────────────────────────────────────────────┘
```

- **L1（实时）**：所有核心游戏逻辑必须在这一层完成，不允许网络调用。
- **L2（近实时）**：统计学习与启发式算法，可异步但必须 <1 帧内完成。
- **L3（云端）**：LLM 生成与复杂推理，仅用于离线预生成或非阻塞运行时增强。

### 1.2 关键数据流

```
玩家行为 → L1 规则引擎 → 游戏状态变更
                ↓ (可选)
         L2 统计适配 → 难度/推荐调整
                ↓ (可选, 高情感节点)
         L3 LLM 生成  → 对话/描述增强 (通过离线预生成库回退)
```

### 1.3 核心设计约束

| 约束 | 说明 |
|------|------|
| 规则确定性 | 相同输入 → 相同输出，配方和 NPC 行为可重现 |
| 离线优先 | LLM 内容在开发期批量预生成，运行时纯查表 |
| 降级策略 | 所有 AI 依赖的功能必须有规则回退方案 |
| 性能边界 | L1 <3ms, L2 <20ms, L3 不阻塞主线程 |

## 2. 多模型 AI 编排方案

### 2.1 模型路由策略

不同开发/运行任务自动路由到最合适的模型：

| 任务类型 | 推荐模型 | 理由 |
|----------|----------|------|
| GDScript 代码生成/重构 | Claude/DeepSeek | 代码理解强，逻辑推理好 |
| 像素美术资源生成 | Stable Diffusion (本地) + GPT 提示词工程 | SD 生成图像，GPT 优化提示词 |
| 多模态理解 (截图分析) | Gemini/GPT-4V | 视觉理解能力 |
| 游戏文案/对话生成 | DeepSeek/GPT | 创意写作 |
| JSON 数据批量生成 | DeepSeek/Claude | 结构化输出稳定 |

### 2.2 技术可行性

- **开发期多模型**：完全可行，通过工具链脚本路由到不同 API
- **运行期多模型**：不推荐，增加延迟和成本，应统一使用单一 API 端点
- **本地 + 云端混合**：SD 本地生图 + 云端 LLM 生成文案，已在项目中部分实现

### 2.3 推荐方案

```
开发期：代码 → DeepSeek | 美术 → SD Local + GPT prompt | 数据 → DeepSeek
运行期：统一 web API 端点 (当前: ark.cn-beijing.volces.com)
```

> 注：当前项目实际使用 `deepseekV4Pro` 作为默认和重试模型，BASE_URL 为 `https://ark.cn-beijing.volces.com/api/v3/chat/completions`。

## 3. 动态配方生成系统 (RecipeGenerator)

### 3.1 核心概念

从"固定配方表"转向"基于标签属性的动态配方匹配"，使任何材料组合都有可能产生结果。

### 3.2 标签本体引擎 (TagOntology)

`data/tag_ontology.json` — 统一物品/NPC/事件的标签字典：

```json
{
  "ingredient_tags": {
    "base": ["grain", "meat", "herb", "fruit", "magic"],
    "element": ["fire", "water", "earth", "air", "light", "dark"],
    "quality": ["fresh", "rotten", "blessed", "cursed", "dragon_touched"],
    "emotion": ["hatred", "love", "fear", "greed", "hope"],
    "process": ["raw", "cooked", "fermented", "burnt", "frozen"]
  },
  "process_tags": ["heat", "stir", "shake", "curse", "bless", "pray", "mock"],
  "container_tags": ["barrel", "grill", "pot", "cauldron"]
}
```

### 3.3 配方生成算法

`scripts/systems/recipe_generator.gd` — 核心方法：

```
try_generate(a_key, b_key, container_type, process) → RecipeResult
```

**四阶段流程**：

1. **标签交集匹配** (`_match_tags`)：两个材料的标签取交集，交集非空才有合成可能
2. **冲突检测** (`_detect_conflicts`)：直接标签互斥、容器-工序不兼容、元素相克
3. **规则表查询** (`_query_rule_table`)：基于 `require_tags` / `forbid_tags` / `process_required` 匹配
4. **结果生成** (`_generate_result`)：生成产品 `item_key`、命名、描述

### 3.4 冲突检测规则

| 冲突类型 | 示例 |
|----------|------|
| 直接标签互斥 | `fire` + `water` → 冲突 |
| 元素相克 | `light` + `dark` → 产物不稳定 |
| 容器-工序不兼容 | `barrel` + `heat` → 无效操作 |
| 祝福/诅咒互斥 | `blessed` + `cursed` → 互相抵消 |

### 3.5 配方规则表 (recipe_rules.json)

```json
{
  "rules": [
    {
      "id": "bread_base",
      "require_tags": ["grain", "flour"],
      "forbid_tags": ["rotten"],
      "process_required": "heat",
      "container": "grill",
      "result": {
        "base_key": "bread",
        "quality_mod": 0,
        "special_effects": []
      }
    }
  ]
}
```

规则匹配优先级：精确匹配 > 模糊匹配 > 默认回退。

### 3.6 与现有系统的对接

- 替换 `CraftSystem.query_recipe()` 的硬编码逻辑
- 保留 `recipes.json` 作为确定性回退
- 动态配方结果写入 `GameManager.discovered_recipes` 字典

### 4.1 核心概念

从预设 NPC 池扩展到特征库驱动的动态生成，使每个客人都有独特的行为模式。

### 4.4 行为权重推导

```
最终行为权重 = 原型基础权重 × 背景调整 × 怪癖叠加
```

例如：矮人战士 (drink=0.4) + 流放贵族 (drink+0.1) + 怕冷 (prefers_warm=drink+0.05) → 最终酒水偏好 0.55

- 保留客人的随机台词池 (`guest_reactions.json`)
- 秘密暴露条件由 `NarrativeManager` 检查

## 5. 剧情编织系统 (StoryWeaver)

### 5.1 核心概念

从线性剧本转向事件触发 + 分支选择的动态叙事，保持故事连贯性的同时增加玩家自由度。

### 5.2 事件触发机制

`scripts/systems/story_weaver.gd` — `_check_trigger()`：

| 触发条件类型 | 示例 |
|--------------|------|
| 天数条件 | `day >= 15 AND day <= 20` |
| NPC 条件 | `ryan.trust > 50` |
| 统计条件 | `total_gold > 200` |
| 标签条件 | `served_item.has_tag("cursed")` |
| 世界条件 | `world_flag("tavern_rumors") == true` |

### 5.3 连续性仲裁 (ContinuityFilter)

确保故事不被破坏的过滤机制：

1. **冷却检查**：同一事件类型 N 天内不重复出现
2. **每 NPC 唯一事件**：同一 NPC 不能同时有两个进行中的事件线
3. **互斥事件**：A 事件发生后 B 事件永久失效
4. **故事线阶段验证**：必须先完成阶段 1 才能触发阶段 2

### 5.4 事件优先级排序

| 优先级 | 类型 | 权重 |
|--------|------|------|
| 100 | 主线高潮事件 | 确保出现 |
| 40 | 角色个人线 | 优先触发 |
| 20 | 世界事件/支线 | 次要填充 |

同优先级按 `last_triggered_time` 升序排列，最久未触发的优先。

### 5.5 故事碎片库 (story_fragments.json)

```json
{
  "fragments": [
    {
      "id": "ryan_truth_01",
      "storyline": "ryan",
      "phase": 1,
      "triggers": {"day_min": 5, "ryan_served_count": 3},
      "nodes": [
        {
          "type": "dialogue",
          "dialogue_file": "ryan_truth_01.dialogue",
          "choices": ["sympathize", "doubt", "ignore"]
        }
      ],
      "branches": {
        "sympathize": {"ryan_trust": 15, "next_fragment": "ryan_truth_02a"},
        "doubt": {"ryan_trust": 5, "next_fragment": "ryan_truth_02b"}
      },
      "continuity": {"cooldown_days": 3, "exclusive_npc": "ryan"}
    }
  ]
}
```

### 5.6 事件总线与影响系统


## 6. 自由合成系统 (状态仿真范式)

### 6.1 范式转变

从"固定枚举"转向"状态仿真"：

| 维度 | 旧范式 (预设成品) | 新范式 (状态仿真) |
|------|-------------------|-------------------|
| 材料 | 固定 item key | SubstanceState 向量 |
| 工序 | 预设路线 | 任意 Process 变换 |
| 产出 | 枚举产品 | 收敛到原型 |
| 自由度 | 有限组合 | 无限探索 |

### 6.2 SubstanceState

`scripts/systems/substance_state.gd` — 物质状态的向量表示：

```gdscript
class SubstanceState:
    var labels: Dictionary        # {"fire": 0.8, "meat": 0.6, "cursed": 0.3}
    var numeric_attrs: Dictionary # {"heat_level": 0.75, "moisture": 0.4, "stability": 0.9}
    var lineage: Array            # ["flour", "cursed_flour"] 成分谱系
    var process_history: Array    # ["heat", "stir", "curse"] 工序历史
    var irreversibles: Dictionary # {"raw->cooked": true} 不可逆标记
```

**关键设计**：标签用 0~1 连续值而非布尔，允许"半熟""微焦"等中间态。

### 6.3 ProcessEngine

`scripts/systems/process_engine.gd` — 工序 = 纯变换函数：

```gdscript
class ProcessEngine:
    # 每个工序是一个 f(SubstanceState) → SubstanceState 的映射
    func apply_heat(state: SubstanceState, intensity: float) -> SubstanceState:
        state.labels["heat"] += intensity * 0.2
        state.numeric_attrs["heat_level"] = clamp(state.numeric_attrs["heat_level"] + intensity, 0, 1)
        # 不可逆：生→熟
        if state.labels.get("raw", 0) > 0.5 and state.numeric_attrs["heat_level"] > 0.3:
            state.irreversibles["raw->cooked"] = true
        # 过火：熟→焦
        if state.numeric_attrs["heat_level"] > 0.8:
            state.labels["burnt"] = state.numeric_attrs["heat_level"] - 0.8
        return state

    func apply_curse(state: SubstanceState, power: float) -> SubstanceState:
        state.labels["cursed"] = clamp(state.labels.get("cursed", 0) + power, 0, 1)
        state.numeric_attrs["stability"] -= power * 0.3
        # 与 blessed 抵消
        if state.labels.get("blessed", 0) > 0:
            var cancel = min(state.labels["blessed"], power)
            state.labels["blessed"] -= cancel
            state.labels["cursed"] -= cancel * 0.5
        return state
```

### 6.4 SubstanceRegistry (量化哈希归并)

解决组合爆炸问题——将连续值离散化，归并相似状态：

```gdscript
class SubstanceRegistry:
    # 量化各维度容差
    const QUANTIZATION = {
        "heat_level": 0.15,    # 相邻 0.15 以内的归并
        "moisture": 0.2,
        "stability": 0.1
    }

    func hash_state(state: SubstanceState) -> int:
        # 将量化后的标签向量哈希为唯一 ID
        return _quantized_hash(state)

    func register(state: SubstanceState) -> String:
        var h = hash_state(state)
        if _registry.has(h):
            return _registry[h]  # 已有归并
        var name = _generate_name(state)
        _registry[h] = name
        return name
```

### 6.5 ProductConverger (产品收敛)

`scripts/systems/product_converger.gd` — 将任意 SubstanceState 收敛到有意义的产品：

```json
// data/product_archetypes.json
{
  "archetypes": [
    {
      "id": "bread_like",
      "target_labels": {"grain": 0.7, "heat": 0.4},
      "forbid_labels": {"cursed": 0.5},
      "quality_tiers": {
        "perfect": {"min_match": 0.9, "max_forbid": 0.1},
        "normal":  {"min_match": 0.6, "max_forbid": 0.3},
        "strange": {"min_match": 0.4, "max_forbid": 0.5},
        "dangerous": {"min_match": 0.2}
      }
    }
  ]
}
```

**四级品质**：
- **perfect**：匹配度 ≥90%，禁止标签 ≤10% → "金黄酥脆的面包"
- **normal**：匹配度 ≥60%，禁止标签 ≤30% → "面包"
- **strange**：勉强可食用但不寻常 → "散发着怪味的面团块"
- **dangerous**：含有危险标签 → "诅咒附着的焦炭"

新发现的原型（未命中任何已知 archetype）触发 `EventBus.emit("new_discovery")`，给予额外奖励。

### 6.6 防循环与防退化

| 机制 | 实现 |
|------|------|
| 工序历史追踪 | `process_history` 数组记录所有工序，重复序列超过阈值拒绝 |
| 不可逆标签 | `raw→cooked→burnt` 单向链，不可回退 |
| 稳定性衰减 | 每次工序消耗 stability，低于 0 时产物变为 "ruined" |
| 工序上限 | 每个 Substance 最多接受 N 次工序（可配置，默认 8） |

## 7. AI 集成运行时代三层架构

### 7.1 L1 实时规则层 (<3ms)

**目标**：所有核心游戏循环必须在这一层完成。

| 子系统 | 实现方式 |
|--------|----------|
| BehaviorTree | 预设行为树，NPC 决策（等待→点单→评价→离开） |
| UtilityEvaluator | 6 因子加权评分选择食物：口味偏好 (0.35) + 价格敏感 (0.25) + 新鲜感 (0.15) + 社交展示 (0.10) + 健康顾虑 (0.08) + 随机噪声 (0.07) |
| TagMatcher | 基于 TagOntology 的标签匹配与冲突检测 |
| RecipeGenerator | 动态配方生成（参见第 3 节） |

```gdscript
# UtilityEvaluator 评分示例
func evaluate_order(customer: DynamicNPC, candidates: Array[FoodItem]) -> FoodItem:
    var best_score = -INF
    var best_item = null
    for item in candidates:
        var score = 0.0
        score += customer.taste_match(item.tags) * 0.35
        score += (1.0 - item.price / customer.max_budget) * 0.25
        score += customer.novelty_bonus(item.key) * 0.15
        score += customer.social_display_score(item) * 0.10
        score += customer.health_concern(item) * 0.08
        score += randf_range(-0.05, 0.05) * 0.07  # 随机噪声
        if score > best_score:
            best_score = score
            best_item = item
    return best_item
```

### 7.2 L2 统计启发层 (<20ms, 异步)

**目标**：基于玩家行为统计的适应性调整。

| 子系统 | 功能 |
|--------|------|
| DifficultyAdapter | 监控玩家成功率、经济趋势，自适应调整订单难度和客人预算 |
| SemanticRecommender | 基于标签相似度推荐未探索的配方方向 |
| StoryBranchSelector | 30 维玩家行为向量 + 离线预生成嵌入，选择最相关的故事分支 |
| TrendAnalyzer | 追踪最近 N 天的经济/菜品/客人趋势，调整生成参数 |

**StoryBranchSelector 嵌入方案**：

```
玩家行为 → 30 维特征向量 (serving_style, favorite_tags, npc_relations...)
         → 余弦相似度匹配离线预计算的 300 个故事分支嵌入
         → 选择 top-3 候选 → L1 优先级排序 → 最终触发
```

### 7.3 L3 云端生成层 (1-3s, 异步)

**目标**：LLM 增强对话和内容生成，仅在必要时触发。

| 子系统 | 触发条件 | 降级策略 |
|--------|----------|----------|
| LLMDialogueBridge | 高情感节点（信任突破阈值、剧情高潮、结局） | 预设对话池回退 |
| EmbeddingRetriever | 离线预计算嵌入索引 | 降级到文本匹配 |
| ContentGenerator | 工具链脚本，开发期批量运行 | 无运行时代码 |

**LLMDialogueBridge 设计**：

```gdscript
class LLMDialogueBridge:
    const COOLDOWN_SECONDS = 30   # 两次 LLM 调用最小间隔
    const MAX_RETRIES = 2
    var _last_call_time = 0
    var _cache = {}

    func generate_dialogue(context: DialogueContext) -> String:
        # 1. 检查缓存
        if _cache.has(context.hash()):
            return _cache[context.hash()]

        # 2. 冷却检查
        if Time.get_ticks_msec() - _last_call_time < COOLDOWN_SECONDS * 1000:
            return _fallback_dialogue(context)  # 降级

        # 3. 异步调用 LLM
        var promise = _call_llm_async(_build_prompt(context))
        await promise.completed

        # 4. 缓存并返回
        if promise.success:
            _cache[context.hash()] = promise.result
            _last_call_time = Time.get_ticks_msec()
            return promise.result
        else:
            return _fallback_dialogue(context)
```

### 7.4 各层职责边界

| 问题 | 所属层级 | 说明 |
|------|----------|------|
| 这个材料组合能否合成？ | L1 | 规则匹配，必须确定性 |
| 这个客人点什么？ | L1 | 效用计算，可重现 |
| 今天的客人难度？ | L2 | 统计自适应 |
| 该触发哪个故事分支？ | L2 + L1 | L2 选候选，L1 定最终 |
| 这句对话怎么写？ | L3 (缓存) + L1 (回退) | 优先缓存，降级到预设 |
| 新配方的名字？ | 离线 L3 | 开发期批量生成到 JSON |

## 8. AI 在动态配方中的价值定位

### 8.1 核心结论

**AI 不参与核心生成引擎**（配方逻辑必须确定性、可重现），但在以下方面有极高价值：

### 8.2 价值分布

| 领域 | 价值占比 | 说明 |
|------|----------|------|
| **命名与描述** | ~60% | LLM 为数千种标签组合生成风味描述 |
| **探索引导** | ~20% | 基于未探索标签组合的模糊提示 |
| **相似推荐** | ~15% | 嵌入向量相似度检索 |
| **核心合成逻辑** | 0% | 规则引擎，AI 无价值 |

### 8.3 命名与描述生成 (RecipeNamer)

`scripts/systems/recipe_namer.gd` — 运行时纯查表系统：

```json
// data/recipe_names.json — 离线批量生成 (2000-5000 条)
{
  "names": [
    {
      "tags": ["grain", "heat", "fresh"],
      "quality": "perfect",
      "name": "金黄酥皮面包",
      "description": "外层金黄酥脆，内里松软温热，麦香四溢",
      "flavor_notes": ["麦香", "微甜", "温暖"]
    },
    {
      "tags": ["meat", "curse", "dark"],
      "quality": "dangerous",
      "name": "怨灵焦肉",
      "description": "表面浮动着暗紫色的诅咒纹路，散发着不祥的气息",
      "flavor_notes": ["焦苦", "寒意", "绝望"]
    }
  ]

```

**离线生成流程**：

```python
# scripts/tools/generate_recipe_names.py (计划)
# 1. 枚举所有合法标签组合 (~2000-5000 种)
# 2. 每种组合 + 品质等级 → LLM prompt
# 3. 批量调用 API 生成 name/description/flavor_notes
# 4. 输出 recipe_names.json
# 5. 运行时: RecipeNamer.lookup(tags, quality) → {name, desc}
```

### 8.4 探索引导

`SemanticRecommender` 分析玩家已发现的配方标签集合，找出未探索但可能产生有趣结果的标签组合：

```gdscript
# 伪代码
func suggest_exploration(discovered_tags: Set, all_tags: Set) -> Array[String]:
    var unexplored = all_tags - discovered_tags
    var candidates = _pairwise(unexplored, discovered_tags)  # 新组合
    # 优先推荐冲突标签对 (可能产生 "strange" 或 "dangerous" 品质)
    candidates.sort_by_conflict_score()
    return _to_vague_hints(candidates[:3])  # 模糊提示，不直接给答案
```

### 8.5 相似推荐

当玩家发现新配方时，推荐配方空间中与之相似的已知/未知配方：

```
用户发现 "诅咒面包" [grain:0.7, heat:0.5, cursed:0.6]
  → 嵌入向量 → 余弦相似度 top-5:
      1. "怨灵烤肉" [meat:0.6, heat:0.5, cursed:0.7] (0.89)
      2. "暗影炖汤" [herb:0.5, heat:0.4, cursed:0.5] (0.82)
      ...
  → 展示推荐（已知配方显示名字，未知用 "???" 提示）
```
