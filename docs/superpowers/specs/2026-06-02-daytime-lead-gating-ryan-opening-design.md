# 白天行动重设：线索门控与莱恩开场重排

> 日期：2026-06-02
> 状态：设计已确认（机制实现延后，本 spec 仅作存档与下次开工依据）
> 目标：消除「Day2 地点凭空冒出」的突兀感，把白天从「跟夜晚不说话的水龙头」改成「给夜晚自由发子弹」的前置环节。

## 1. 问题诊断

玩家反馈：Day2 突然冒出几个调查地点很突兀；进一步暴露出白天采集玩法「无聊又突兀」。拆开看，「无聊」和「突兀」是同一个病根——**白天和夜晚之间没有信息回路**——的两个症状。

具体四个毛病（基于 `day_map_view.gd` / `day_map_system.gd` / `locations.json`）：

1. **采集是零风险兑换机**：每个点固定花体力、固定产出，无变量、无取舍、不会失败。→「无聊」。
2. **白天与夜晚单向脱钩**：玩家在不知道今晚谁来、点什么的情况下盲目囤货。两个阶段只靠「库存」一根管子单向相连。→「突兀」的深层来源。
3. **采集 ≈ 商店**：两个功能重叠的材料来源，无差异化。
4. **叫 Map 实为滚动按钮列表**：空间、距离、未知都没被玩法利用。

### 1.1 莱恩调查线的三个叠加结构问题

以莱恩 Day2 调查线（佣兵告示板 → 废弃矿道 → 公会柜台）为最小复现：

- **纯日期闸门**：地点唯一出现条件是 `dayMin: 2`（`day_map_system.gd:51`）。剧情里没有「为什么是现在」。
- **时序倒置（最致命）**：本该当钩子的莱恩台词「明天我要跟血斧小队去地牢」在 **Day2 夜**，而调查地点在 **Day2 早**。玩家在还没有任何理由关心血斧小队时，先去调查了血斧小队。动机和行动装反了。
- **整条线一次性全曝光**：`get_locations()` 只按 `dayMin` 过滤、不看 `requiresFlag`/`requiresRead`（`day_map_system.gd:48-53`），三个调查点同时出现；点矿道/公会被 `visit()` 拦「还没找到线索」。地图跑在剧情前面。

> 关键观察：门控逻辑**已写一半**——`visit()`（`day_map_system.gd:56-67`）已检查 `requiresFlag`/`requiresRead`，只是 `get_locations()` 显示时不门控。这让本 spec 的机制改动比看上去小得多。

## 2. 北极星原则

> **白天只负责喂入信息与利害，永远不替夜晚预定结果。**

守住这条，白天再有结构也不会跟夜晚的沙盘自由矛盾；违反它，白天再自由也会把夜晚压成过场动画。

推论：
- 白天给夜晚的是两种**输入**——知识/利害（调查）+ 物料（采集/采买）——从不决定夜里怎么用。
- 体力是一份稀缺预算，在「了解」和「准备」之间竞争；这个取舍是白天反复出现、有后果的核心决策。
- 回路闭合：**白天发现 → 夜晚选择的重量 → 次日世界的回应**。

## 3. 范围

### 3.1 本 spec 覆盖（In scope）

- **A. 剧情重排（纯对话内容）**：莱恩 Day1.pre / Day1.post / Day2.pre 重写，使血斧钩子在 Day1 离店时落下，Day2 早的调查不再突兀。注意 A 在 Day1.post 设置的 `ryan_warhammer_lead` 须由阶段1 注册并被 `get_locations()` 消费——A 与阶段1 应一起落地，不单独发 A（否则该 flag 是无人读取的孤儿，门控仍走旧的裸 dayMin）。
- **B. 线索门控机制（两阶段）**：让地点「逐格出现」、第一格由前一晚的对话 flag 点燃。
  - 阶段1（纯逻辑）：`get_locations()` 门控 + 线索 flag 接线。
  - 阶段2（UI）：地图列表分区 + 新线索提示。

### 3.2 非目标（Out of scope，留作未来）

- **白天整体重设为「知道 vs 准备」天平**：本 spec 只修莱恩线 + 通用门控机制，不重做采集/商店的差异化与软情报系统。那是更大的一笔，单独立项。
- **调查产出从「正解」改为「更难的取舍」**：当前调查链终点仍是「替代委托」这一近似正解。把它改成有代价的取舍（替代委托报酬低/莱恩永远升不了白银阶）是独立的剧本+数据工作，本 spec 不含，仅在 §7 标记为后续。
- 采集点本身的玩法深化（产出选择、风险）。
- 任何 C 方向（风险探索 / roguelite）的内容。

## 4. 第一部分：剧情重排（对话改动）

### 4.1 设计意图

莱恩线现在缺的不是机制，是「第一段」——头两次出场要把他写成**具体到让人牵挂、又隐隐让人担心**的人；血斧钩子要落在玩家已经在意的人身上；血斧线索要在**前一晚离店**时落下（满足「线索从前一晚来」）。

- **依恋钩子**（B 基底）：莱恩给老家妹妹寄钱攒学费，梦想不是虚荣而是责任。
- **忧虑种子**：「一趟顶半年的大委托」——他有非赚不可的理由去赌命。
- **枢轴台词**：Day2 的「老板，你看我……能行吧?」把球踢给玩家，是整场戏的道德支点。

### 4.2 Day1 · pre（B 基底）

```
~ start
莱恩: 老板，一杯麦芽酒，便宜的就行，我得省着点。
莱恩: 今天也单挑了三只史莱姆！赏金我都寄回老家了——妹妹还等着攒学费呢。
莱恩: 所以我得快点升上去。白银阶队伍的赏金，是现在的十倍。
莱恩: 等我进了白银阶，接一趟顶半年的大委托，妹妹就不用再等了。
=> END
```

### 4.3 Day1 · post（离店时落下线索）

无论那杯成没成，他都会去——计划是计划——只是语气分两种。**血斧小队**第一次被点名发生在这里（离店一刻），玩家揣着这句话过夜。

```
~ start

if serve_result == "success"
    莱恩: 呼——这杯下肚，浑身是劲！老板，你这手艺，够白银阶的队伍抢着来喝了！
    莱恩: 对了——告示板上有支「血斧小队」在招北矿道的护送，赏金抵我跑三个月！
    莱恩: 我明天就去报名。攒够这一笔，妹妹的学费就有着落了——明天见，老板！
    set aff_ryan = 5
    set ryan_warhammer_lead = true
    => END
else
    莱恩: 唉，今天手头紧，这杯就够了。
    莱恩: ……告示板上那支「血斧小队」在招北矿道护送，赏金高得吓人，我明天去试试。
    莱恩: 总得拼一把，不然妹妹的学费猴年马月才攒得齐。明天见。
    set ryan_warhammer_lead = true
    => END
```

> 注：原 Day1.post 仅有成功/失败两句反馈，现新增「离店落线索」的叙事职责。
> 注：「白银区」一词已废弃——只用现有设定「白银阶」（rank，非地点）。不引入计划外世界观。

### 4.4 Day2 · pre（忧虑发芽，钩子落在已在意的人身上）

血斧不再是陌生名词，而是「那个给妹妹寄钱的孩子明天要去的地方」。若玩家 Day2 早已查到送死局，反讽在此满弦。

```
~ start
莱恩: 老板，来份烤肉，再来杯麦芽酒——明天就出发，得吃饱点。
莱恩: 血斧小队答应带我去北矿道，护送一趟抵我三个月赏金。妹妹的学费，这下稳了。
莱恩: 我知道北矿道凶险……可这种机会，错过了不知要等到哪年。
莱恩: （搓了搓手）说实话，心里有点没底。老板，你看我……能行吧?
=> END
```

夜晚的回应（壮胆送行 / 醒神劝退 / 沉默照做）由现有 `ryan_day2.post` 的 `ryan_informed` / `ryan_has_alternative` 分支承接，本 spec 不改 post。

## 5. 第二部分：线索门控机制

### 5.1 阶段1：纯逻辑（低风险，不碰 UI 结构）

**目标**：地点逐格正确出现；第一格（告示板）由前一晚的线索 flag 点燃，不再靠裸 `dayMin`。

改动：

1. **注册线索 flag**：在 `_fresh_narrative_vars()`（`game_manager.gd:768`，项目明示的真相源）与 `narrative_manager.load_npc_data()` 的默认值中**同步**加入 `ryan_warhammer_lead: false`（两处必须一致，见 `game_manager.gd:767` 注释）。
2. **对话点燃**：见 §4.3 的 `set ryan_warhammer_lead = true`。
3. **告示板改门控**：`locations.json` 中 `mercenary_board` 加 `requiresFlag: ryan_warhammer_lead`。保留 `dayMin: 2` 作下限，但真正的闸门是 flag——两者都满足才显示（避免 flag 被剧情提前置真时地点早于预期出现）。
4. **`get_locations()` 门控显示**：仿照 `visit()` 的检查，过滤掉 `requiresFlag` 未满足、`requiresRead` 未读的地点。约 6 行。
5. **跨天 flag 注入（唯一架构坑）**：`DayMapSystem._flags` 每天 `start_day()` 清空，用于当天内的 `mine_clue` 等临时解锁；而 `ryan_warhammer_lead` 须**跨天存活**，它住在 `narrative.dialogue_vars`（持久、且进存档）。按中介模式（CLAUDE.md 硬规则 4），`DayMapSystem` 不直接读 narrative——由 `GameManager` 把持久线索 flag **喂进** `get_locations()`（加 flags 参数，或 `start_day()` 注入）。`get_locations()` 须同时查「当天临时 flag」+「持久线索 flag」两个来源。

**阶段1 完成后效果**：地点已能逐格正确出现，「时序倒置 / 全曝光 / 纯日期闸门」功能上全部修好。地点仍在同一列表里，但行为正确。

### 5.2 阶段2：UI（可隔离的后半段）

逻辑改完后，地点会「正确地」逐格出现，但玩家仍感觉不到「这是一条故事线」（告示板与菌菇林地混在一个列表）。UI 三件事，按投入从小到大：

1. **列表分区（必做）**：`_build_location_ui()` / `_reload_location_ui()` 按 `type`（`gather` / `investigation`）分组渲染：
   ```
   —— 采集 ——        (常驻、可重复)
     菌菇林地 / 暗河沿岸 / 葡萄藤架 / 农庄磨坊
   —— 线索调查 ——    (事件驱动，挂在活跃线索下)
     《血斧小队的委托》
       └ 佣兵告示板  [1体力]
   ```
   需要在 `locations.json` 给地点加 `type` 字段（采集点 `gather`，调查点 `investigation` + `lead` 归属）。
2. **新线索提示（建议）**：复用现成的顶栏 `DocumentsBtn` + `DocumentOverlay`，线索挂进文档系统冒红点，不新造 UI。
3. **体力天平可视化（可选）**：做了 ① 的分区后基本自带——两段同屏可见即让「采集与调查抢同一份体力」的取舍有体感。顶栏体力数字够用。

> 阶段2 是面子不是地基。可先做阶段1、跑起来亲眼看手感，再决定 UI 做多重。

## 7. 验证方式

无 test runner；验证 = headless 跑 + 走核心路径（参见项目 headless 测试约定）。

- **对话（A）**：`test_ryan_delivery` / `test_ryan_actions` 回归不破；手动走 Day1 夜 → 确认离店台词点名血斧、`ryan_warhammer_lead` 置真。
- **机制阶段1**：headless 断言——`ryan_warhammer_lead=false` 时 `get_locations()` 不含 `mercenary_board`；置真后含之，且矿道/公会在前置满足前不出现（逐格）。存档往返后 flag 保持（`test_save_roundtrip`）。
- **机制阶段2**：手动进 DayMap，确认采集与线索分区显示、新线索有提示。
- 全流程 0 error / 0 warning。
