# 冲击魔法 — 力度合成能力商店化（设计）

- 日期：2026-06-05
- 分支：`feat/seasoning-system`
- 状态：设计已确认，待写实现计划

## 一句话

把实验场景里"两材料大力对撞就合成"的力度机制，做成商店分阶段购买的**魔法能力**，买了之后在正式 Tavern 营业里能徒手砸出成品、绕过厨具容器；砸的力度有窗口，砸太狠出**劣质**成品，并借此把一直被无视的 `quality` 字段全面接进上菜收益。

## 背景 / 现状

- 正式 Tavern (`scripts/ui/bar_workspace.gd` + `scenes/ui/Tavern.tscn`) 已是物理工作台：材料是 `DeskItem` 物理体，容器是炖锅 `Pot` / 烤架 `Grill` / 酒桶 `Brewery`（`kitchen_container.gd`、`brewery.gd`）。厨具合成 = 把材料拖进容器进料口 → 搅拌/摇/煎到位 → 出成品。配方走 `craft.query_recipe(container, ingredients)`。
- 力度合成只活在 `scripts/ui/experimental_tavern_view.gd` 沙盘：两个 `DeskItem` 相对速度 ≥ `MERGE_SPEED_THRESHOLD`(350) 撞一起 → `get_combine_result(a,b)` 出结果。**未接入正式循环。**
- 商店在 DayMap (`scripts/ui/day_map_view.gd`)，现有两块：材料、配方解锁。解锁状态走 `craft.unlock_recipe / is_recipe_unlocked`，存档持久化（`game_manager.gd` save/load）。
- `DeskItem.quality`（`scripts/test/desk_item.gd:46`，字符串，默认 `"normal"`）和酒桶产出的 `good/normal` 品质 **当前在上菜逻辑 `_on_serve_requested` 中完全没被读取** —— 收益只看"订单匹配 + 物品定价"。属于死管道。

## 目标 / 非目标

**目标**
1. 商店新增"技法"区，分两档售卖力度合成能力（魔法）。
2. 买了之后，正式 Tavern 里两材料对撞可直接出对应容器类配方，绕过容器。
3. 力度窗口：太轻不合成、窗口内满质、太狠出劣质。
4. 把 `quality` 全面接入上菜收益（poor/normal/good 三档各有金币与声望差异），让"砸劣质"真有代价。
5. 解锁状态与"已购买能力"进存档。

**非目标（YAGNI）**
- 不动烤架（grill）：它是"按压煎"，没有"砸"的语义，自然排除。
- 不新增"劣质/废料"物品：劣质用 `quality="poor"` 表达，复用现成字段。
- 不改容器的"按天解锁"逻辑（`workspace.is_container_unlocked`）——与购买能力平行共存。
- 不重写厨具合成：力度合成是**新增旁路**，厨具保持原样。

## 已知影响（务必标注）

- **酒桶经济会变**：`brewery` 一直产 `good/normal` 但被无视；本功能让 quality 生效后，摇得好（good）将首次真正给到收益加成。这是预期的正向变化，但属于经济侧改动，验收时需观察一遍酒桶产出收益是否仍合理。

## 核心机制

### 1. 力度窗口（砸合成）

正式 Tavern 里，两个材料 `DeskItem` 碰撞，按相对速度判定：

| 相对速度 | 结果 |
|---|---|
| `< MERGE_LOW`（≈350，沿用现值） | 不合成，白撞（已有行为） |
| `MERGE_LOW … MERGE_HIGH` | 正常成品，`quality = "normal"` |
| `> MERGE_HIGH`（≈1000，可调） | **砸烂**：成品 `quality = "poor"`，舞台浮字提示"手重了" |

- `MERGE_LOW` / `MERGE_HIGH` 放进数据文件（如 `data/craft_style_thresholds.json` 或新 `data/slam.json`）热调，不写死。
- 反馈走现有"三出口"里的舞台浮字（`tavern_view` 的 StageCaption API），不复活底部 HUD。

### 2. 配方映射（砸 → 容器配方）

砸合成查的是**容器配方表**（让砸真能替代炖锅/酒桶），但只认已解锁能力的 container 类：

- **双材料配方**（肉汤=生肉+麦芽、香料红酒=葡萄+草药…）：两个不同材料对撞，食材集 = 这对。
- **单材料配方**（麦芽酒=麦芽、葡萄酒=葡萄…）：撞两个**相同**材料 → 出 2 份成品（沿用实验场景同种碰撞产双份的逻辑）。
- 新增查询 `craft.find_slam_recipe(keys: Array) -> Dictionary`：遍历 `recipes`，找 `container ∈ 已解锁砸的类` 且 `ingredients`（排序）匹配 `keys`（排序）的产物，返回 `{ "product": key, "container": c }`，无匹配返回 `{}`。
- **双重门禁仍生效**：若配方本身 `requires_purchase` 且未解锁（如肉汤、香料红酒），即使有冲击魔法也不能砸出——能力解锁的是"方法"，配方解锁的是"知道这道菜"，两者独立叠加。

### 3. 分阶段解锁

| 档 | 名字 | 解锁范围 | 价格 |
|---|---|---|---|
| `slam_pot` | 冲击魔法·炖锅 | 砸出所有 `container:"pot"` 配方 | 60 金 |
| `slam_barrel` | 冲击魔法·酒桶 | 砸出所有 `container:"barrel"` 配方 | 100 金 |

两档独立购买，无先后强制（酒桶可先买）。名字/价格写进 JSON，随时可调。

### 4. quality → 上菜收益

`_on_serve_requested` 在订单**匹配成功**时按成品 quality 调整收益（失败不涉及）：

| quality | 金币 | 声望 |
|---|---|---|
| `good` | `floor(price × 1.5)` | +3 |
| `normal` | `price`（现值） | +2（现值） |
| `poor` | `floor(price × 0.5)` | +0 |

倍率与声望增量做成常量（或小 JSON），便于调参。`normal` 维持现有数值，保证非劣质路径行为不变。

## 组件改动清单（按依赖顺序）

1. **`data/shop.json`** — 新增 `abilities` 段：`[{ "key": "slam_pot", "name": "冲击魔法·炖锅", "price": 60 }, { "key": "slam_barrel", "name": "冲击魔法·酒桶", "price": 100 }]`。
2. **数据：力度阈值** — `MERGE_LOW`/`MERGE_HIGH` 与 quality 收益倍率落到数据文件。
3. **`scripts/systems/shop_system.gd`** — 加载 `abilities`，`get_ability_price(key)`、`get_ability_name(key)`、`get_abilities()`。
4. **`scripts/systems/craft_system.gd`**
   - `var unlocked_slam_categories: Array`、`is_slam_unlocked(category)`、`unlock_slam(category)`。
   - `find_slam_recipe(keys) -> Dictionary`（含 requires_purchase 校验、单/双材料处理）。
   - key↔container 映射：`slam_pot→"pot"`、`slam_barrel→"barrel"`（一处常量）。
5. **`scripts/game_manager.gd`**
   - `buy_ability(key) -> bool`（扣钱 + `craft.unlock_slam(category)`，已购返回 false，与 `buy_recipe_unlock` 同构）。
   - `_on_serve_requested`：成功路径按 quality 调整金币/声望。
   - save/load：序列化 `craft.unlocked_slam_categories`（与 `unlocked_recipes` 同处）。
6. **`scripts/ui/bar_workspace.gd`** — 从 `experimental_tavern_view` 移植碰撞合成段：
   - `DeskItem.body_entered` 连接 + 相对速度判定（去重、`call_deferred` 出物理回调）。
   - 命中 `find_slam_recipe`；门禁不通过则不合成。
   - 力度窗口决定 `quality`；生成成品（双材料一份 / 单材料两份），保留动量。
   - 砸烂时触发舞台浮字。
7. **`scripts/ui/day_map_view.gd`** — 商店 UI 加"技法"区，列出能力行：名称 / 价格 / 购买按钮 / 已购态（已购买变灰显示"已掌握"）。复用现有 `_build_*_rows` 风格。
8. **`scripts/test/desk_item.gd`** — 如需，补 `quality` 的便捷设置（多数情况直接赋值即可）。

## 测试策略（headless）

参考 memory「Godot headless 测试」与现有 `scripts/test/` 套路：

- **CraftSystem 单测**：`find_slam_recipe` 对双材料/单材料/未解锁类/requires_purchase 未解锁/无匹配各分支；`is_slam_unlocked`/`unlock_slam` 幂等。
- **ShopSystem 单测**：abilities 加载、价格/名称查询、未知 key。
- **GameManager 单测**：`buy_ability` 扣钱与重复购买；quality→收益三档金币/声望数值；save→load 往返保住已购能力。
- **力度窗口**：阈值分类（low/window/high → 不合成/normal/poor）纯逻辑可抽函数单测。
- 物理碰撞与 UI 走查：编辑器内走核心路径（TitleScreen→DayMap 买能力→Tavern 砸合成→上菜→Ledger），0 错 0 警。

## 开放参数（实现时定，可热调）

- `MERGE_HIGH` 具体值（建议 ~1000，需手感走查）。
- quality 收益倍率（good ×1.5 / poor ×0.5 为起始值）。
- 能力价格（60 / 100 为起始值）。
