# 核心机制

> 本文定义当前已实现或已接入主流程的玩法规则。旧的固定合成槽、CraftStation 按钮手势、SeasoningZone 三状态调味台不是当前 Tavern 主流程。

---

## 一、昼夜循环

游戏以天为单位推进，当前目标仍是 30 天结构。

| 阶段 | 场景 | 玩家行为 |
|------|------|----------|
| 白天准备 | `DayMap` | 选择地点、采集、商店、NPC 对话、整理携带材料 |
| 夜晚营业 | `Tavern` | 在物理吧台上取材、加工、上菜 |
| 打烊结算 | `LedgerScreen` | 查看金币、声望和剧情结果 |
| 结局 | `EndingScreen` | 第 30 天后展示结局 |

`DayCycleSystem` 负责日期和阶段，`GameManager` 负责场景注册、切换和数据传递。

---

## 二、Tavern 物理工作面

当前 Tavern 的主交互由 `BarWorkspace` 驱动。

### 2.1 取材

1. `BarWorkspace` 读取 `GameManager.inventory`。
2. 非 product 物品填入底部 `ShortcutBar` 的 10 个槽位。
3. 玩家左键点击槽位，在鼠标位置生成 `DeskItem`。
4. `DeskItem` 根据 `items.json` 与 `item_physics_profiles.json` 应用颜色、质量、摩擦、弹性和碰撞形状。

### 2.2 拖拽

- `DragController` 用 PinJoint2D 钉住物理体。
- 可拖动材料、成品、酒桶、锅/烤架和勺子。
- 松手后物体继续由物理引擎模拟。
- `DeskItem.is_held` 标记抓取状态，烤架按压煎制依赖该标记。

### 2.3 上菜

1. 玩家把 product 类型的 `DeskItem` 拖到 `CustomerDropArea`。
2. 松手时 `BarWorkspace._try_serve()` 读取拖拽期间的上菜速度。
3. 调用 `GameManager.request_serve(item_key, craft_style_data)`。
4. `GuestSystem` 结算正确/错误订单，`EconomySystem` 更新金币声望。
5. `NarrativeManager` 读取配方记忆和 L3 风格，决定 NPC 反馈。

## 三、配方系统

配方定义在 `data/recipes.json`，当前结构为：

```json
{
  "wine": {
    "name": "葡萄酒",
    "ingredients": ["grape"],
    "container": "barrel",
    "memory_for": { "mira": "day4_road_story" }
  }
}
```

`CraftSystem.query_recipe(container, ingredients)` 使用 `container + sorted(ingredients)` 查询产物。材料顺序不影响结果。

### 当前配方

| 容器 | 成品 | 材料 |
|------|------|------|
| 酒桶 | `ale_beer` 麦芽酒 | `ale` |
| 酒桶 | `wine` 葡萄酒 | `grape` |
| 酒桶 | `herb_tea` 草药茶 | `herb` |
| 酒桶 | `herbal_ale` 草药麦酒 | `ale` + `herb` |
| 酒桶 | `spiced_wine` 香料红酒 | `grape` + `herb` |
| 烤架 | `bread` 面包 | `dough`（面粉搅拌成面团） |
| 烤架 | `meat_cooked` 烤肉 | `meat_raw` |
| 烤架 | `meat_sand` 肉夹面包 | `dough_meat`（面粉搅拌成面团，面团+生肉砸成肉夹生面团） |
| 炖锅 | `meat_stew` 肉汤 | `meat_raw` + `ale` |
| 炖锅 | `herb_broth` 草药清汤 | `herb` + `ale` |
| 炖锅 | `malt_porridge` 麦芽粥 | `flour` + `ale` |

价格当前从 `data/items.json` 的 product `price` 读取。

## 四、容器机制

### 4.1 酒桶 `Brewery`

位置：`Tavern/BarWorkspace/World/Brewery`，脚本仍在 `scripts/test/brewery.gd`。

规则：

- 材料进入桶口并达到接收速度阈值后才被接收。
- 多材料可在短窗口内累积，统一查询配方。
- 抓住酒桶左右摇晃，`BrewShakeMeter` 根据水平速度方向翻转计数。
- 摇晃满足阈值后，从 `OutputAnchor` 弹出成品。
- 摇晃质量影响 product 的 `quality`，当前为 `normal` / `good`。

### 4.2 烤架 `KitchenContainer(container_key="grill")`

规则：

- 可烤物品进入 `SearZone` 且被玩家抓着时，按 `heat_rate` 累积单一熟度。
- 肉类使用 `MeatDoneness` 记录单一熟度；两半视觉共享同一个熟度颜色，不再要求翻面。
- 物品离开烤区时定稿：
  - 生：保持原物品。
  - 熟：按 `query_recipe("grill", [item_key])` 变成成品。
  - 焦：变成 `meat_burnt` 或 `bread_burnt`。
- 已经是 product 的普通物品不会被当作新配方输入；`bread`、`meat_cooked` 可继续煎到焦糊。

### 4.3 炖锅 `KitchenContainer(container_key="pot")`

规则：

- 材料进入 `Intake` 后加入内部 `CookStationState` 并销毁桌面物体。
- 锅默认冻结为静态，抓起时临时解冻，松手后重新冻结，避免被勺子推跑。
- 勺尖在锅内的搅拌判定区移动时累积进度。
- 达到 `required_stir` 后，按锅内材料查询配方并从 `OutputAnchor` 弹出成品。
- 配方不匹配时清空当前材料并输出日志，不生成物品。

## 五、物品物理与 profile

`DeskItem` 是材料和成品的统一物理体。

| 数据 | 来源 | 用途 |
|------|------|------|
| 物品名/颜色/价格/类型 | `data/items.json` | 显示、订单、经济 |
| physics profile | `data/item_physics_profiles.json` | mass、friction、bounce、damp、gravity |
| collision profile | 同上 | rect/circle/capsule 碰撞形状 |
| feedback profile | 同上 | impact 声音、粒子、shake 预留 |

当前常用 profile：

- `round_light`：葡萄，轻、圆、弹。
- `heavy_dull`：生肉，重、钝、低弹。
- `soft_stable`：面粉，软、稳定、不弹。
- `default`：麦芽、草药等默认物品。

## 六、客人系统

`GuestSystem` 当前仍是单客服务模式。

| 行为 | 结果 |
|------|------|
| 正确上菜 | 获得金币和声望，记录服务结果 |
| 错误上菜 | 负面反馈，扣声望 |
| 耐心耗尽 | 客人离开并扣声望 |

客人显示由 `TavernView` 更新，真实上菜入口由 `GameManager.request_serve()` 统一处理。

## 七、L3 动作风格

当前 L3 MVP 只按上菜速度分类：

| 风格 | 含义 |
|------|------|
| 温柔 | 慢、轻、谨慎 |
| 平静 | 默认中间档 |
| 粗鲁 | 快、重、冲撞感 |

阈值来自 `data/craft_style_thresholds.json`。`CraftStyleSystem.classify()` 返回中文风格名。

叙事使用方式：

1. 配方可通过 `memory_for` 标记某 NPC 记忆，例如 wine 对 Mira。
2. NPC 在 `npcs.json` 中配置 `preferred_styles` 和 `disliked_styles`。
3. `NarrativeManager.resolve_serve_style()` 同时看 L2 配方记忆与 L3 风格。
4. 匹配 preferred 可讲出故事并加信任；匹配 disliked 则触发负面反馈。

当前已验证的竖切片是 Mira Day 4 / wine。

## 八、白天活动

`DayMap` 从 `data/locations.json` 加载地点。当前地点类型包括采集、对话和商店相关入口。

白天阶段的结果会影响库存、NPC 变量和进入 Tavern 时的可用材料。

---

## 九、对话与叙事

重要 NPC 对话文件命名：

```text
dialogue/{npc_id}_day{N}.pre.dialogue
dialogue/{npc_id}_day{N}.post.dialogue
```

- `pre`：上菜前对话或点单。
- `post`：上菜后反馈，读取 `serve_result`、配方记忆、风格和剧情变量。

当前 NPC 内容：

| NPC | 天数 | 状态 |
|-----|------|------|
| Ryan | Day 1-3 | 宿命样板线 |
| Mira | Day 4、Day 12 | 商人线；Day 4 已接入 wine + L3 |

## 十、经济、商店与库存

- `EconomySystem` 管理金币和声望。
- product 价格在 `items.json`。
- `ShopSystem` 读取 `shop.json`，处理购买材料和配方解锁。
- `GameManager.inventory` 是当前库存主入口。
- 当前 `BarWorkspace` 把快捷栏作为材料源，库存扣减仍是后续可细化点。

---

## 十一、香料系统状态

`SeasoningSystem` 和 `data/seasonings.json` 仍存在，但旧的 `SeasoningZone` 调味流程已经不属于当前 Tavern 主交互。后续如要恢复香料，应优先按物理工作面模型设计：香料作为物理物品或成品属性进入 `BarWorkspace`，不要恢复固定槽位调味台。

## 十二、验证路径

项目没有通用自动测试 runner。主要验证方式：

```text
TitleScreen -> DayMap -> Tavern -> LedgerScreen -> DayMap
```

三容器改动至少验证：

- 酒桶单材料和双材料配方。
- 酒桶摇晃不足和摇够两种路径。
- 烤架生/熟/焦路径。
- 炖锅双材料搅拌产出。
- 成品拖入 `CustomerDropArea` 后正确结算。
- Mira Day 4 / wine 的 L3 风格分支不回归。
